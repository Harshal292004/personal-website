## Introduction

This is my first blog post, and the goal here is simple to document thoughts, learnings, and mistakes while building things on the web. No tutorials pretending everything worked on the first try, no motivational fluff. Just real notes from working with modern web tooling.

The web ecosystem moves fast. React abstractions evolve, Next.js keeps changing how we think about rendering, routing, and data fetching, and every project teaches something new whether you want it or not.

## Why React and Next.js

React makes UI state predictable. Components are just functions of state and props, and once that mental model clicks, everything else builds on top of it.

Next.js sits on top of React and solves problems that every serious project eventually runs into:

- Routing without external libraries
- Server-side rendering and static generation
- File-based organization that scales decently
- API routes living alongside frontend code

It removes a lot of boilerplate, but it also introduces opinions. Learning when to follow them and when to bypass them is part of the process.

## What I Learned Building This

Even for something as basic as a blog, there are decisions everywhere:

- How content is stored (markdown vs CMS vs database)
- How metadata is handled (frontmatter, tags, series)
- Build-time vs runtime rendering
- Keeping components clean instead of shoving logic everywhere

Markdown with frontmatter keeps things boring and predictable. Git becomes the database. Version history becomes content history. That tradeoff works well for personal writing.

## Mistakes and Friction Points

A few things that were annoying and worth remembering:

- Over-abstracting too early makes refactoring worse, not better
- Styling consistency matters more than fancy animations
- Performance problems usually come from data flow, not rendering
- Good folder structure saves more time than clever code

None of these are new insights, but ignoring them always costs time.

## What’s Next

This blog is going to be used for:

- Writing about things I break and then fix
- Explaining concepts to future me
- Dumping notes that don’t belong in code comments

The idea is consistency, not perfection. If something is confusing today, it probably deserves a post.

## Closing

This is the starting point. The stack will change, opinions will change, and some posts will age badly. That’s fine. The point is to ship, write, and move forward instead of waiting for things to feel “ready”.

More posts coming.
